# Procyon v0.1.0 — Process Guardian Design Spec

## Overview

Procyon is a lightweight process protection tool for long-running ML workloads on shared Linux servers. It prevents duplicate process launches, protects registered processes from being killed by agents or other users, and auto-restarts crashed processes.

**Version**: 0.1.0
**Target platform**: Ubuntu 24.04, kernel 6.8, Python 3.12, stdlib only
**Primary user**: Claude Code agents (agent-first); human as secondary
**Deployment**: Single shared server (10.11.219.245), user pthahnix

## Motivation

Real incident: a stuck Claude Code sub-agent spawned 4 duplicate training processes into an already-running job slot (`rvq_full_nopca`), all writing to the same checkpoint directory. Separately, manual kill of apparent duplicates has previously destroyed legitimate DataLoader workers mid-training.

Procyon solves this from the moment an ML process starts, leaving a clear, machine-readable record that prevents any future agent or human from misjudging or misoperating.

## Concurrency & File Locking

All read-modify-write operations on `registry.json` use `fcntl.flock()` (exclusive lock) to prevent race conditions from concurrent `register`/`unregister` calls. If the lock cannot be acquired within 5 seconds, the command fails with error code `LOCK_TIMEOUT`.

Lock files (`procyon.lock`) use atomic write (write to temp file + `os.rename()`) to prevent partial reads.

## CLI Commands

All commands output JSON by default. Add `--pretty` for human-readable output.

### `procyon register`

```
procyon register --name <name> --pid <pid> --cmd "<cmd>" [--checkpoint_dir <path>]
```

- Check lock file exists → PID alive → refuse (anti-duplicate)
- If lock exists but PID dead → clean stale lock → proceed
- Write lock file to `<checkpoint_dir>/procyon.lock` or `~/.procyon/locks/<name>.lock`
- Write entry to `~/.procyon/registry.json`
- Output: `{"status": "registered", "name": "...", "pid": ...}`

### `procyon unregister`

```
procyon unregister --name <name>
```

- Remove from registry.json
- Delete lock file
- Output: `{"status": "unregistered", "name": "..."}`

### `procyon status`

```
procyon status [--pretty]
```

- Default: JSON array of all registered processes
- Each entry: `{name, pid, cmd, checkpoint_dir, started, uptime, alive, protected}`
- `--pretty`: human-readable table
- Checks if each PID is actually alive

### `procyon kill`

```
procyon kill --name <name>
```

- **With TTY (human)**: display process info, require typing job name to confirm
- **Without TTY (agent/script)**: refuse entirely, output JSON error
- Flow: confirm → unregister (remove from registry) → SIGTERM → wait 10s → SIGKILL
- Unregistering first ensures watchdog will NOT restart the killed process

### `procyon watch`

```
procyon watch [--interval 30]
```

- Daemonize via double-fork (fork → setsid → fork again → parent exits) to fully detach from terminal
- On start: check `~/.procyon/watchdog.pid` — if PID alive, refuse; if PID dead, clean stale PID file and proceed
- Poll registry every `interval` seconds
- If registered PID is dead:
  - Check if completion marker exists in checkpoint_dir (default: `checkpoint_final.pt`, configurable via `--done-marker` on register/run) → clean exit, auto-unregister, skip restart
  - Otherwise → restart using saved `cmd` → update registry with new PID → log event
- Log to `~/.procyon/watchdog.log`
- PID file at `~/.procyon/watchdog.pid`
- Responds to SIGTERM gracefully: unlink PID file, exit
- `procyon watch --stop`: send SIGTERM to running watchdog via PID file

### `procyon run`

```
procyon run --name <name> [--checkpoint_dir <path>] [--done-marker <filename>] -- <cmd...>
```

Wrapper mode (non-invasive integration):

1. Fork child process to execute `<cmd>`
2. In child (before exec): set process name via `ctypes.CDLL("libc.so.6").prctl(15, b"procyon:<name>")` — non-fatal if fails, just logs warning
3. Signal protection strategy: the **parent** process intercepts SIGTERM/SIGHUP and refuses to forward them to the child. The child retains its own signal handlers (e.g., PyTorch graceful checkpoint saving). Direct signals to the child PID from other users are blocked by Layer 2 (AppArmor).
4. Auto-register with child's PID
5. Forward SIGINT to child (allow Ctrl+C)
6. On child exit: auto-unregister
7. Output: `{"status": "started", "name": "...", "pid": ..., "cmd": "..."}`

### `procyon issue`

```
procyon issue --title <title> --body <body> [--priority low|medium|high] [--tag bug|feature|improvement]
```

- Write to `PROCYON/issues/YYYY-MM-DD_NNN_<slug>.md`
- File format with frontmatter:

```markdown
---
title: <title>
priority: medium
tag: bug
date: 2026-03-23T04:30:00
author: claude-code
status: open
---

<body>
```

- Output: `{"status": "created", "path": "issues/..."}`

## Error Handling

All errors use a consistent JSON envelope:

```json
{"status": "error", "code": "<ERROR_CODE>", "message": "Human-readable description"}
```

Error codes:

| Code | Command | Meaning |
|---|---|---|
| `DUPLICATE_PROCESS` | register | Lock exists and PID is alive |
| `NOT_FOUND` | unregister, kill | Job name not in registry |
| `ALREADY_DEAD` | kill | PID in registry is no longer alive |
| `NO_TTY` | kill | No interactive terminal detected |
| `LOCK_TIMEOUT` | register, unregister | Could not acquire registry file lock within 5s |
| `WATCHDOG_RUNNING` | watch | Watchdog already running (PID alive) |
| `INVALID_ARGS` | any | Missing or invalid arguments |

## Protection Layers

### Layer 1: Soft Protection (procyon.py)

**Signal protection** (wrapper mode `procyon run`):
- **Parent** intercepts SIGTERM/SIGHUP and refuses to forward to child
- Child process keeps its own signal handlers intact (preserves graceful shutdown in training frameworks like PyTorch Lightning)
- SIGINT forwarded (allows human Ctrl+C)
- SIGKILL cannot be blocked (known limitation, covered by Layer 2)

**Process identification**:
- `prctl(PR_SET_NAME)` via `ctypes` (stdlib) sets process comm to `procyon:<name>`
- Non-fatal: if ctypes call fails, logs warning and continues
- Visible in `ps`, `top`, `htop` — other users see the name and know not to touch it

**Agent protection**:
- `procyon kill` refuses execution without TTY
- `procyon status` JSON includes `"protected": true`
- CC skill mandates: check status before any training launch

### Layer 2: AppArmor (install.sh)

One-time setup via `install.sh` (requires sudo):

1. Generate AppArmor profile `/etc/apparmor.d/procyon.training`
2. Profile transition mechanism: `procyon run` writes target profile name to `/proc/self/attr/exec` before `execvp()`, so the child process starts confined
3. Profile rules:
   - Processes registered by procyon run under `procyon_training` profile
   - `signal(receive)` restricted: only same-uid processes via procyon.py can send signals
   - Other users' `kill` commands blocked by AppArmor, logged to audit log
4. Load profile with `aa-enforce`
5. Create `~/.procyon/` directory structure

**Graceful degradation**: if AppArmor is not installed or profile not loaded, `procyon run` logs a warning and continues with Layer 1 only. Protection is reduced but training is not blocked.

**Limitation**: root can `aa-disable procyon.training` to bypass, but this is a highly deliberate action.

### Layer 3: PID Namespace (future, not in v0.1.0)

Reserved for future implementation. Would use `unshare --pid` to make processes invisible to other users entirely.

## Skill Design

### `skill/procyon-usage.md` — Daily Usage SOP

Teaches CC:
1. Before launching training: always `procyon status` to check for conflicts
2. Preferred launch method: `procyon run --name <name> -- <cmd>` (wrapper mode)
3. Never use raw `kill`/`pkill` on training processes
4. Never use `procyon kill` (will be rejected without TTY anyway)
5. Parse JSON output for decision-making
6. Use `procyon issue` to report problems, not self-fix procyon.py

### `skill/procyon-iterate.md` — Iteration SOP

Teaches CC the upgrade workflow (triggered by human command, executed under human supervision):

```
1. Read issues/ with status: open
2. git checkout main && git pull
3. git checkout -b procyon/issue-NNN-<slug>
4. Modify procyon.py and/or tests/
5. Run smoke test + e2e test (python3 -m pytest tests/)
6. All pass → commit, push, create PR (body references issue)
7. Fail → fix or abandon branch, file issue explaining failure
8. Report to human: summarize changes, link to PR
9. Human reviews on GitHub, says "merge"
10. CC merges PR
11. On main: bump VERSION, tag v{VERSION}, close resolved issues
12. Deploy: sync to server
```

**Rollback**: if post-merge issues found, `git checkout v{previous}-stable -- procyon.py` on main.

## Registry Format

`~/.procyon/registry.json`:

```json
{
  "processes": {
    "rvq_full_pca": {
      "pid": 3203377,
      "cmd": "python3 scripts/train_rvq.py --checkpoint_dir ...",
      "checkpoint_dir": "/data/pthahnix/MeshLex-Research/checkpoints/rvq_full_pca",
      "started": "2026-03-21T19:25:00",
      "registered_by": "procyon_run",
      "done_marker": "checkpoint_final.pt"
    }
  },
  "version": "0.1.0"
}
```

**Version migration**: when procyon reads a registry with a version older than its own, it migrates the format in-place and logs the migration. Unknown fields are preserved (forward-compatible).

## Key Constraints

- **Python stdlib only** — no external dependencies (ctypes is stdlib)
- **Single file** — `procyon.py` contains all runtime logic; `install.sh` is a one-time setup script, not part of the runtime
- **Non-invasive** — wrapper mode requires zero changes to training scripts
- **Fail-safe** — if procyon crashes, training is unaffected
- **Agent-first** — JSON default output, TTY-gated kill, machine-readable everything
- **Non-interactive TTY detection** — `kill` refuses without TTY (core agent protection)

## Testing Strategy

`tests/test_procyon.py`:

- Lock file CRUD: create, read, stale detection, cleanup
- Registry CRUD: register, unregister, concurrent access
- Anti-duplicate: refuse when PID alive, allow when PID dead (stale)
- TTY detection: mock isatty() for kill command
- Signal protection: verify SIGTERM/SIGHUP ignored in wrapper mode
- Watchdog: mock process death, verify restart logic
- Issue creation: verify file format and frontmatter
- JSON output: verify all commands produce valid JSON

## Integration Example

### Wrapper mode (preferred, zero changes to training script):

```bash
python3 procyon.py run --name rvq_full_pca \
  --checkpoint_dir /data/pthahnix/MeshLex-Research/checkpoints/rvq_full_pca \
  -- python3 scripts/train_rvq.py --checkpoint_dir ... --epochs 15
```

### Script integration mode (2-line addition):

```python
# At top, after args parsed:
import subprocess, os, sys
subprocess.run(["python3", "/home/pthahnix/PROCYON/procyon.py", "register",
                "--name", f"rvq_{args.exp_name}",
                "--pid", str(os.getpid()),
                "--cmd", " ".join(sys.argv),
                "--checkpoint_dir", args.checkpoint_dir], check=False)

# In finally block:
subprocess.run(["python3", "/home/pthahnix/PROCYON/procyon.py", "unregister",
                "--name", f"rvq_{args.exp_name}"], check=False)
```
