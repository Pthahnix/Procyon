## Overview

## Issue #001: save_registry TOCTOU Race

**Problem:** `save_registry()` opens the registry file with `'w'` mode, which truncates it immediately — before `fcntl.flock()` acquires the exclusive lock. Two concurrent writers can both truncate the file, then one writes while the other's data is lost.

**Fix:** Replace `open('w')` with `os.open(O_RDWR | O_CREAT)` to get a file descriptor without truncation. Acquire `LOCK_EX` on the fd, then `ftruncate(fd, 0)` + `lseek(0)` + `write()`. This ensures the lock is held before any data is destroyed.

```python
def save_registry(reg):
    path = _registry_path()
    fd = os.open(str(path), os.O_RDWR | os.O_CREAT, 0o644)
    try:
        fcntl.flock(fd, fcntl.LOCK_EX)
        os.ftruncate(fd, 0)
        os.lseek(fd, 0, os.SEEK_SET)
        os.write(fd, json.dumps(reg, indent=2).encode())
    finally:
        os.close(fd)
```

**Risk:** Low. The function signature and behavior are unchanged from the caller's perspective.

## Issue #002: Watchdog Lock File Not Updated After Auto-Restart

**Problem:** `watchdog_loop()` updates the registry's PID and `started` timestamp after restarting a crashed process, but never calls `write_lock()` to update the lock file. The lock file retains the dead PID until the next stale-lock cleanup.

**Fix:** Add a `write_lock()` call in the auto-restart branch, immediately after `save_registry()`:

```python
write_lock(name, new_proc.pid, proc["cmd"], checkpoint_dir)
```

**Risk:** Minimal. `write_lock()` uses atomic tmp+rename, so no corruption risk.

## Issue #003: File Permissions 0775 → 0755

**Problem:** `procyon.py` has `0775` permissions (group-writable). Should be `0755` to prevent group members from modifying the guardian.

**Fix:** `chmod 755 procyon.py` — a one-time operational fix. No code changes required. The correct permission is committed via `git add` (git records the executable bit).

**Risk:** None.

---

## Issue #004: daemonize() Leaks Inherited File Descriptors

**Problem:** After double-fork daemonization, only `sys.stdin/stdout/stderr` are redirected to `/dev/null`. All other inherited file descriptors (fd 3+) remain open. This is a well-known daemon anti-pattern that can cause resource leaks and prevent file cleanup.

**Fix:** Before redirecting stdio, close all fds from 3 to the system maximum:

```python
import resource
maxfd = resource.getrlimit(resource.RLIMIT_NOFILE)[1]
if maxfd == resource.RLIM_INFINITY:
    maxfd = 1024
os.closerange(3, maxfd)
```

Uses `resource.getrlimit(RLIMIT_NOFILE)` for the upper bound (more accurate than hardcoding). `os.closerange()` silently skips already-closed fds.

**Risk:** Low. Only affects the daemon process after fork — the parent (user's shell) is unaffected.

## Issue #005: kill --yes Bypasses TTY Check

**Problem:** `procyon kill --name <job>` rejects with `NO_TTY` error when called from Claude Code's non-interactive shell. Claude Code is the primary operator of Procyon but cannot stop jobs it launched.

**Fix:** Add a `--yes` flag to the `kill` subcommand:

```python
p_kill.add_argument('--yes', action='store_true',
                    help='Skip TTY check and confirmation (for non-interactive use)')
```

When `--yes` is passed:
- **Skip:** TTY check (`sys.stdin.isatty()`)
- **Skip:** Interactive name-typing confirmation prompt
- **Keep:** `NOT_FOUND` check (always runs)
- **Keep:** `ALREADY_DEAD` check (always runs)
- **Keep:** SIGTERM → wait → SIGKILL sequence (unchanged)

When `--yes` is NOT passed: behavior is completely unchanged from v0.1.0.

**Usage:**
```bash
python3 procyon.py kill --name COLOSSUS --yes
```

**Risk:** Low. Default behavior unchanged. `--yes` is opt-in. Safety checks (process existence, liveness) still apply.

## Testing Strategy

Each fix gets targeted tests:

- **#001:** Test that `save_registry` works correctly (roundtrip). Concurrency test is difficult in unit tests but the fix is a well-known pattern.
- **#002:** Test that watchdog restart updates the lock file PID.
- **#003:** No test needed (operational fix).
- **#004:** Test that `daemonize()` closes inherited fds (verify fd 3+ are closed after call, using `os.pipe()` to create a known fd before daemonize).
- **#005:** Test `kill --yes` bypasses TTY check. Test that without `--yes`, NO_TTY error still occurs in non-interactive shell.
